Ниже — полный калькулятор ключевых метрик MySQL, которые вы будете использовать на дашборде. Каждая метрика расписана по единому шаблону:

Что показывает → Как вычисляется → Нормальные значения → Что делать при отклонении

Весь материал разбит на 4 логических блока.

Блок 1. Доступность и общая нагрузка (верхняя панель дашборда)

1.1 Uptime (время жизни сервера)
Параметр		Значение
Что показывает		Сколько секунд (или дней) работает MySQL с момента последнего перезапуска
Ключ Zabbix		mysql.uptime
Единицы измерения	секунды (на дашборде перевести в дни/часы)
Норма			от дней до месяцев. Короткий uptime (< 1 часа) возможен только после планового обслуживания
Тревога			Если uptime неожиданно сбросился — MySQL перезапустился (сбой, OOM Killer, ручной перезапуск)

1.2 Queries per second (QPS)
Параметр		Значение
Что показывает		Среднее количество SQL-запросов в секунду. Главный измеритель нагрузки на сервер
Ключ Zabbix		mysql.questions.rate (или вычисляемое)
Единицы измерения	запросов/сек
Норма			Зависит от мощности сервера. Важна динамика, а не абсолютное число. Резкий рост/падение — сигнал
Тревога			Резкое падение → возможно, приложение не работает. Резкий рост → проверяйте, не атака ли или дефектный запрос

1.3 Slow queries (медленные запросы)
Параметр		Значение
Что показывает		Количество запросов, превысивших порог long_query_time (обычно 1-10 секунд)
Ключ Zabbix		mysql.slow_queries или mysql.slow_queries.rate
Единицы измерения	количество (или в секунду)
Норма			0 в идеале. Допустимо очень редкое появление (< 1 в час)
Тревога			Любое появление — повод включить slow_query_log и найти виновный запрос

Блок 2. Соединения (предупреждение о перегрузке)

2.1 Connection utilization (процент использованных соединений)
Параметр		Значение
Что показывает		Какую долю от максимально допустимых соединений (max_connections) сейчас занято
Формула			last(//mysql.threads_connected) / last(//mysql.max_connections) * 100
Единицы измерения	%
Норма			< 70% в спокойное время, < 85% в пик
Тревога			> 85% → предупреждение. > 95% → высокий риск отказа в новых подключениях (приложение начнёт падать)

2.2 Aborted connections (оборванные соединения)
Параметр		Значение
Что показывает		Количество соединений, которые не были установлены корректно
Ключ Zabbix		mysql.aborted_connects или mysql.aborted_connects.rate
Норма			0 или очень низкое значение. Немного может быть при нормальной работе
Тревога			Резкий рост → проверяйте: атака перебором паролей? Неправильные пароли в приложении? Сетевые проблемы?

Блок 3. InnoDB (самое важное для производительности)

3.1 Buffer pool hit ratio (попадания в кэш) — ГЛАВНАЯ МЕТРИКА
Параметр		Значение
Что показывает		Какой процент запросов к данным MySQL обслужил из оперативной памяти (быстро), а какой — с диска (медленно)
Формула			100 - (last(//mysql.innodb_buffer_pool_reads) / last(//mysql.innodb_buffer_pool_read_requests)) * 100)
Единицы измерения	%
Норма			≥ 99% — отлично. 95-98% — терпимо, но уже проблема. < 95% — критично
Что означает		Если значение падает → данные не помещаются в RAM, MySQL вынужден постоянно читать с диска. Это в 10-100 раз медленнее
Что делать		Увеличить innodb_buffer_pool_size (доступная RAM). Оптимизировать запросы (нужно меньше данных таскать). Если не помогает — добавлять физической памяти

3.2 Buffer pool utilization (заполнение буферного пула)
Параметр		Значение
Что показывает		Насколько заполнен буферный кэш InnoDB
Формула		(100 - (last(//mysql.innodb_buffer_pool_pages_free) / last(//mysql.innodb_buffer_pool_pages_total)) * 100)
Единицы измерения	%
Норма			70-90% — оптимально. > 95% — кэш переполнен. < 50% — возможно, памяти выделено слишком много
Взаимосвязь		Работает в паре с hit ratio. Если utilization > 95% и hit ratio падает → памяти точно не хватает

3.3 Physical reads per second (чтение с диска)
Параметр		Значение
Что показывает		Скорость физического чтения страниц с диска (когда данных нет в кэше)
Ключ Zabbix		mysql.innodb_buffer_pool_reads.rate
Норма			Стремится к нулю. Допустимы редкие пики при старте или больших отчётах
Тревога			Если стабильно > 100-1000 в секунду и hit ratio падает → проблема с размером кэша

Блок 4. Репликация (только для slave-серверов)

4.1 Seconds behind master (отставание реплики)
Параметр		Значение
Что показывает		На сколько секунд slave отстаёт от master (задержка применения изменений)
Ключ Zabbix		mysql.replication.seconds_behind_master
Норма			0 в идеале. Допустимо 1-10 секунд в нормальной работе
Тревога			Стабильно > 60 секунд → проблема. > 300 секунд → критично (при переключении потеряете данные)
Причины отставания	Слабый slave-сервер, тяжёлые запросы на slave, сетевые задержки, длительные блокировки

4.2 Slave thread status (жива ли репликация)
Параметр		Значение
Что показывает		Работают ли два потока репликации: I/O thread (получение данных) и SQL thread (применение)
Ключи			mysql.replication.io_thread_running, mysql.replication.sql_thread_running
Норма			Оба YES
Тревога			Любой из них NO → репликация остановлена (например, дубликат ключа или потеря связи)
Сводный чек-лист для дашборда (порядок создания)
№	Блок		Что вывести				Тип виджета			Приоритет
1	Доступность	Uptime					Singlestat (в днях)		🔴 Обязательно
2	Нагрузка	QPS + Slow queries			График (2 линии)		🔴 Обязательно
3	Соединения	Connection utilization (%)		Singlestat с цветом		🔴 Обязательно
4	InnoDB		Hit ratio (%)				Singlestat (верхняя панель)	🔴 КРИТИЧНО
5	InnoDB		Hit ratio + Utilization			График (2 линии)		🔴 Обязательно
6	InnoDB		Physical reads per second		График				🟡 Важно
7	Репликация	Seconds behind master (если slave)	График				🟡 Важно
8	Репликация	Thread status (если slave)		Singlestat или текст		🟡 Важно

Готовые формулы для Zabbix (вставил — работает)
Все формулы используют сокращение // для текущего хоста.

Название метрики	Формула для вычисляемого элемента
Hit Ratio		(100 - (last(//mysql.innodb_buffer_pool_reads) / last(//mysql.innodb_buffer_pool_read_requests)) * 100)
Utilization		(100 - (last(//mysql.innodb_buffer_pool_pages_free) / last(//mysql.innodb_buffer_pool_pages_total)) * 100)
Connection utilization	last(//mysql.threads_connected) / last(//mysql.max_connections) * 100
Теперь у вас есть полный, структурированный справочник. Выводите на дашборд в указанном порядке — и вы будете видеть всё состояние MySQL с одного взгляда.
