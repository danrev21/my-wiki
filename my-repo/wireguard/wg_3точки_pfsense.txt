Исходные данные:
1. Домашняя сеть home1 192.168.0.0/24 и домашняя сеть home2 192.168.1.0/24, находятся физически в разных локация и у разных провайдеров.
2. В home1 развернута лаборатория на базе pfsense:
wan - 192.168.0.101/24
lan - 172.16.0.1/24
wg opt - 10.45.1.1/24
3. На pfsense уже установлен wireguard, и для доступа из home1 к сети lan за pfsense уже настроена впн сеть 10.45.1.0/24.
4. Данные по ОС, интерфейсам и ip адресам wireguard:
wg opt на pfsense-10.45.1.1
home1 - ubuntu 22.04, eth0, wg0 - 10.45.1.2;
home2 - ubuntu 22.04, enp3s0, wg0 - 10.45.1.3;
облачный сервер - ubuntu 22.04, eth0, wg0 - 10.45.1.4;
iphone-10.45.1.5

Измененить сеть wireguard 10.45.1.0/24 таким образом, чтобы:
1. Использовать облачный сервер (10.45.1.4) как relay/посредника.
2. Обеспечить доступ из home1 и home2 к сервисам в сети lan.
3. Выполнив вышеописанные действия, обеспечить доступ с мобильного телефона iphone к сервисам в сети lan, как в пределах home2 так и вне её.
Необходимые мне данные:

2. Изменение настроек в pfsense, в связи с добавлением сети home2 и облачного сервера
3. Конфигурация на iphone

-----------------ответ

Отличная задача по построению отказоустойчивой и гибкой VPN-инфраструктуры. Ваше текущее "звездообразное" подключение всех узлов напрямую к pfSense работает, но требует наличия публичного IP у домашнего шлюза и не позволяет подключаться к home2 извне.

Предлагаемая схема с облачным сервером в качестве хаба (hub) решает эти проблемы. Все домашние сети и мобильные устройства будут подключаться к этому центральному узлу, который будет выступать в роли ретранслятора трафика. Это позволит iPhone подключаться к сервисам в home2 как изнутри ее сети (по локальному IP), так и из любой точки мира через интернет.

1. Общая архитектура и IP-план
Мы оставляем существующую сеть WireGuard 10.45.1.0/24, но меняем логику маршрутизации. Теперь все пиры (home1, home2, iphone) будут подключаться к облачному серверу (10.45.1.4). Сервер будет перенаправлять трафик между ними.

Облачный сервер (Hub): 10.45.1.4/24. Имеет публичный IP, слушает на порту 51820.

Home1 (pfSense): 10.45.1.1/24. Подключается к облачному серверу.

Home1 (Ubuntu): 10.45.1.2/24 (остается без изменений, не участвует в изменениях).

Home2 (Ubuntu): 10.45.1.3/24. Подключается к облачному серверу.

iPhone: 10.45.1.5/32. Подключается к облачному серверу.

2. Конфигурационные файлы WireGuard
A. Конфигурация облачного сервера (релей/хаб)
Этот сервер является центральным узлом. На нем включается IP-форвардинг и настраиваются правила iptables для маршрутизации трафика между пирами и включения доступа к их локальным сетям.

Файл: /etc/wireguard/wg0.conf

[Interface]
Address = 10.45.1.4/24
PrivateKey = <SERVER_PRIVATE_KEY>
ListenPort = 51820
# Включаем форвардинг и разрешаем трафику проходить через этот сервер
PostUp = sysctl -w net.ipv4.ip_forward=1
# Разрешить входящий трафик через VPN-интерфейс
PostUp = iptables -A FORWARD -i %i -j ACCEPT
# Разрешить исходящий трафик через VPN-интерфейс
PostUp = iptables -A FORWARD -o %i -j ACCEPT
# Удалить правила при отключении
PostDown = iptables -D FORWARD -i %i -j ACCEPT
PostDown = iptables -D FORWARD -o %i -j ACCEPT

### Пиры (Peer) ###

# Home1 (pfSense)
[Peer]
PublicKey = <PFSENSE_PUBLIC_KEY>
AllowedIPs = 10.45.1.1/32, 172.16.0.0/24
# PersistentKeepalive нужен, если pfSense за NAT
PersistentKeepalive = 25

# Home2 (Ubuntu Server)
[Peer]
PublicKey = <HOME2_PUBLIC_KEY>
AllowedIPs = 10.45.1.3/32, 192.168.1.0/24
PersistentKeepalive = 25

# iPhone
[Peer]
PublicKey = <IPHONE_PUBLIC_KEY>
AllowedIPs = 10.45.1.5/32

B. Конфигурация Home2 (Ubuntu 22.04)
Этот сервер в домашней сети home2 подключается к облачному серверу.

Файл: /etc/wireguard/wg0.conf

[Interface]
Address = 10.45.1.3/32
PrivateKey = <HOME2_PRIVATE_KEY>
# Включаем форвардинг и NAT для доступа к локальной сети home2
PostUp = sysctl -w net.ipv4.ip_forward=1
PostUp = iptables -A FORWARD -i %i -j ACCEPT
PostUp = iptables -t nat -A POSTROUTING -s 10.45.1.0/24 -o enp3s0 -j MASQUERADE
PostDown = iptables -D FORWARD -i %i -j ACCEPT
PostDown = iptables -t nat -D POSTROUTING -s 10.45.1.0/24 -o enp3s0 -j MASQUERADE

[Peer]
PublicKey = <SERVER_PUBLIC_KEY>
Endpoint = <SERVER_PUBLIC_IP>:51820
AllowedIPs = 10.45.1.0/24, 172.16.0.0/24
PersistentKeepalive = 25

Объяснение: AllowedIPs указывает, что через этот туннель мы можем достичь 172.16.0.0/24 (сеть за pfSense/home1). Правило MASQUERADE на enp3s0 позволяет устройствам из home2 и из VPN-сети (10.45.1.0/24) обращаться к сервисам в локальной сети home2, используя IP самого Ubuntu-сервера.

C. Конфигурация iPhone
Конфигурация создается через официальное приложение WireGuard. Самый простой способ — сгенерировать конфиг с ключами и импортировать его по QR-коду.

Содержимое конфигурационного файла:

[Interface]
PrivateKey = <IPHONE_PRIVATE_KEY>
Address = 10.45.1.5/32
DNS = 8.8.8.8  # Опционально, если нужен DNS через VPN

[Peer]
PublicKey = <SERVER_PUBLIC_KEY>
Endpoint = <SERVER_PUBLIC_IP>:51820
AllowedIPs = 10.45.1.0/24, 172.16.0.0/24, 192.168.1.0/24
PersistentKeepalive = 25

Важно: AllowedIPs на iPhone не должен содержать 0.0.0.0/0, иначе весь интернет-трафик телефона пойдет через облачный сервер. Мы указываем только те сети, к которым нужен доступ: саму VPN-сеть (10.45.1.0/24) и ваши локальные сети (172.16.0.0/24 и 192.168.1.0/24).

3. Изменения в настройках pfSense
Изменение конфигурации WireGuard:

Перейдите в VPN > WireGuard > Tunnels.

Отредактируйте существующий туннель wg_opt (или создайте новый).

В поле Peers измените пир "Облачный сервер":

Public Key: Вставьте публичный ключ вашего облачного сервера.

Endpoint: Укажите публичный IP-адрес и порт облачного сервера.

Allowed IPs: Добавьте 10.45.1.4/32, 10.45.1.3/32, 10.45.1.5/32, 192.168.1.0/24. Здесь важно перечислить все подсети и адреса, которые находятся за туннелем. Не используйте 0.0.0.0/0, чтобы не сломать обычный интернет-доступ.

Включите Persistent Keepalive (например, 25).

Статические маршруты:

Перейдите в System > Routing > Static Routes.

Добавьте маршрут для сети 192.168.1.0/24 (home2). В качестве Gateway выберите созданный вами интерфейс WireGuard.

(Опционально) Добавьте маршрут для 10.45.1.3/32 и 10.45.1.5/32 с тем же шлюзом. Это повысит надежность.

Правила файрвола:

Убедитесь, что на интерфейсе WireGuard (WG_OPT) есть правило, разрешающее любой трафик из источника * в назначение *. Без этого пакеты будут блокироваться.

Для доступа из home2 к сервисам за pfSense (172.16.0.0/24) дополнительные правила не нужны, так как трафик будет идти по туннелю. Однако, если вы хотите ограничить доступ, можно создать правило на интерфейсе LAN (или OPTx, где находится 172.16.0.0/24), разрешающее доступ только с IP-адресов VPN (10.45.1.0/24).

4. Проверка и отладка
После применения всех настроек выполните проверки:

На облачном сервере:

bash
sudo wg show
Убедитесь, что со всеми пирами (home1, home2, iphone) установлен handshake и есть трафик.

С home2 (Ubuntu):

bash
ping 10.45.1.1   # Проверка связи с pfSense
ping 172.16.0.1  # Проверка доступа к локальному шлюзу pfSense
С iPhone:

Находясь в home2 (подключившись к ее Wi-Fi), включите VPN и попробуйте открыть в браузере http://172.16.0.1 (или IP любого сервиса в home1).

Выйдите из home2, переключитесь на мобильные данные, снова включите VPN и повторите попытку. Доступ должен сохраниться.

Если что-то не работает, последовательно проверьте:

Правильность ключей (PublicKey/PrivateKey).

Маршруты: с устройства должен быть путь до целевой сети.

Файрволы: не блокируют ли они трафик на интерфейсах WireGuard.

Правила NAT: особенно на стороне home2, чтобы сервисы могли "видеть" входящие запросы из VPN.
